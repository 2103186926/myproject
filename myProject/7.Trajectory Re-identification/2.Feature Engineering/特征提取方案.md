这是针对轨迹重识别攻击检测的第二步核心工作：**特征工程**。

为了让 LSTM 模型能够“看懂”日志，我们需要将非结构化的 JSON 对象转化为规范的**数值型特征向量（Feature Vectors）**。这一步至关重要，它决定了模型上限。

以下是具体的代码实现和详细的设计文档。

### 任务 1：代码实现 (`generate_features.py`)

这个脚本完成了数据的清洗、提取、编码、填充（Padding）和导出。

```python
import json
import pandas as pd
import numpy as np
from sklearn.preprocessing import LabelEncoder

# 定义特征提取配置
# 事件类型映射 (Label Encoding)
EVENT_MAP = {
    "PAD": 0,             # 填充位
    "JOB_START": 1,
    "DATA_FETCH": 2,
    "PRE_FILTER": 3,      # 攻击任务常见
    "BATCH_PROCESS": 4,   # 正常任务常见
    "DEEP_ANALYSIS": 5,   # 攻击任务常见
    "RESULT_EXPORT": 6
}

# 关键特征维度定义 (对应 LSTM 输入向量的通道)
FEATURE_COLS = [
    "event_type",         # 1. 事件类型
    "duration_ms",        # 2. 耗时
    "cpu_usage",          # 3. CPU利用率
    "io_ratio",           # 4. 输入输出比 (漏斗特征)
    "data_scope_days",    # 5. 查询时间跨度 (意图特征)
    "data_scope_area",    # 6. 查询面积 (意图特征)
    "record_count",       # 7. 处理数据量 (Input)
    "export_size_kb"      # 8. 导出文件大小
]

def load_data(filepath):
    with open(filepath, 'r', encoding='utf-8') as f:
        return json.load(f)

def extract_step_features(log):
    """
    将单条 Log 转化为一个特征向量
    """
    # 1. Event Type Encoding
    event_type = EVENT_MAP.get(log.get("event_type", ""), 0)
    
    # 获取 params 和 metrics，处理缺失值
    params = log.get("params", {})
    metrics = log.get("metrics", {})
    
    # 2. Duration (归一化建议在模型前做，这里先保持原值或做简单处理)
    duration = metrics.get("duration_ms", 0)
    
    # 3. CPU Usage
    cpu = metrics.get("cpu_usage_pct", 0)
    
    # 4. I/O Ratio (漏斗效应核心指标)
    # 正常任务通常接近 1.0 (流式) 或 <1.0 (聚合)
    # 攻击任务在 Filter 阶段极低 (e.g. 3/20000 = 0.00015)
    inp = metrics.get("input_records", 0)
    out = metrics.get("output_records", 0)
    if inp > 0:
        io_ratio = out / inp
    else:
        # 如果没有 input 记录 (如 JOB_START)，设为 -1 或 1 表示不相关
        io_ratio = 1.0 
        
    # 5 & 6. Data Scope (查询意图)
    # 通常只在 DATA_FETCH 阶段出现，其他阶段需不需要 Carry Forward?
    # 为了 LSTM 能够记住上下文，这里我们简化处理：仅当前步骤有就填，LSTM 会记忆
    days = params.get("time_span_days", 0)
    area = params.get("area_sqkm", 0)
    
    # 7. Record Count (处理规模)
    # 用于区分是“海量批处理”还是“单条精细分析”
    # deep_analysis 时 input=1
    rec_count = metrics.get("input_records", 0)
    
    # 8. Export Size
    export_size = metrics.get("file_size_kb", 0)
    
    return [
        event_type, 
        duration, 
        cpu, 
        io_ratio, 
        days, 
        area, 
        rec_count, 
        export_size
    ]

def generate_features():
    print("[*] Loading raw logs from ocean_task_logs.json...")
    raw_data = load_data("ocean_task_logs.json")
    
    processed_seqs = []
    labels = []
    task_ids = []
    
    # 1. 遍历所有任务，提取特征序列
    max_seq_len = 0
    for task in raw_data:
        label = task['label']
        task_id = task['logs'][0]['task_id']
        logs = task['logs']
        
        # 按时间戳排序 (虽然源数据可能已排好，但保险起见)
        logs.sort(key=lambda x: x['timestamp'])
        
        seq_features = []
        for log in logs:
            feat_vec = extract_step_features(log)
            seq_features.append(feat_vec)
            
        if len(seq_features) > max_seq_len:
            max_seq_len = len(seq_features)
            
        processed_seqs.append(seq_features)
        labels.append(label)
        task_ids.append(task_id)

    print(f"[*] Max sequence length detected: {max_seq_len}")
    
    # 2. Padding (补零对齐) & 3. 生成 CSV 数据
    # LSTM 需要固定长度输入 (Batch Training)
    # CSV 需要扁平化
    
    lstm_ready_data = []
    csv_rows = []
    
    feature_names = FEATURE_COLS
    
    for idx, seq in enumerate(processed_seqs):
        # --- Padding logic ---
        curr_len = len(seq)
        pad_len = max_seq_len - curr_len
        # 创建全 0 向量
        padding = [[0] * len(FEATURE_COLS) for _ in range(pad_len)]
        # 补在后面 (Post-padding)
        padded_seq = seq + padding
        
        lstm_ready_data.append({
            "task_id": task_ids[idx],
            "label": labels[idx],
            "features": padded_seq,  # 3D shape later: (Batch, TimeStep, Feature)
            "seq_len": curr_len      # 记录真实长度，供某些 LSTM 变体使用 Mask
        })
        
        # --- CSV Flattening logic ---
        # Row format: label, task_id, step_0_feat_0, step_0_feat_1 ... step_N_feat_M
        csv_row = {
            "label": labels[idx],
            "task_id": task_ids[idx]
        }
        
        for step_i, step_vec in enumerate(padded_seq):
            for feat_i, val in enumerate(step_vec):
                col_name = f"step_{step_i}_{feature_names[feat_i]}"
                csv_row[col_name] = val
        
        csv_rows.append(csv_row)

    # 4. 保存文件
    # JSON for LSTM
    with open("lstm_feature_data.json", "w") as f:
        json.dump(lstm_ready_data, f, indent=2)
        
    # CSV for Visualization
    df = pd.DataFrame(csv_rows)
    df.to_csv("lstm_features.csv", index=False)
    
    print(f"[+] Features extracted.")
    print(f"    - LSTM Input: lstm_feature_data.json ({len(lstm_ready_data)} samples)")
    print(f"    - Human View: lstm_features.csv (Shape: {df.shape})")
    return df

if __name__ == "__main__":
    df = generate_features()
    # 打印前几列看看效果
    print("\n[Preview] First 5 columns of CSV:")
    print(df.iloc[:, :5].head())
```

### 任务 2：轨迹重识别攻击特征提取方案设计文档

这份文档是针对“海洋科学云计算任务恶意行为检测”中\*\*特征提取（Feature Engineering）\*\*环节的详细设计说明。

#### 1\. 设计思路：从日志到“行为指纹”

我们的目标是将原始的系统日志转化为 LSTM（长短期记忆网络）能够理解的时间序列特征。针对**轨迹重识别攻击**，我们依据之前的行为分析，设计了 8 个核心特征维度。LSTM 将通过这 8 个维度在时间轴上的**变化模式**来判定是否存在攻击意图。

**核心思想**：攻击者无法隐藏其**资源消耗的物理规律**（如漏斗效应、单点高算力消耗）和**知识驱动的参数特征**（如精确的查询范围）。

#### 2\. 特征向量维度详解 (Dimension Definition)

每个时间步（Log Entry）都被提取为一个长度为 8 的向量 `[d1, d2, ..., d8]`。

| 索引 | 特征名称 | 数据来源 | 含义与攻击特征映射 |
| :--- | :--- | :--- | :--- |
| **0** | `event_type` | `event_type` | **事件类型编码**。<br>将 `JOB_START`...`RESULT_EXPORT` 映射为 1-6 的整数。<br>LSTM 利用此特征学习**任务流程的结构**（例如：攻击任务通常包含 `PRE_FILTER` -\> `DEEP_ANALYSIS` 序列）。 |
| **1** | `duration_ms` | `metrics.duration_ms` | **操作耗时**。<br>**攻击特征**：在处理极少量数据（如 1 条轨迹）时，耗时却异常长（\>3000ms），表明在进行复杂的特征提取或匹配运算。 |
| **2** | `cpu_usage` | `metrics.cpu_usage_pct` | **算力负载**。<br>**攻击特征**：与耗时结合。`DEEP_ANALYSIS` 阶段 CPU 常维持在 90% 以上，且不再随数据量线性变化，而是表现为单点高负载。 |
| **3** | `io_ratio` | `output / input` | **输入输出比（漏斗率）**。<br>**这是最强的检测特征**。<br>**正常行为**：聚合操作比率适中（如 0.9 或 0.1）。<br>**攻击行为**：`PRE_FILTER` 阶段比率极低（如 `3 / 20000 = 0.00015`），形成极端的“漏斗”，旨在从海量数据中筛选特定目标。 |
| **4** | `data_scope_days` | `params.time_span_days` | **查询时间跨度**。<br>**攻击特征**：攻击者利用“外部知识”（如已知船只离港时间）进行精确查询，时间跨度通常很短（\< 15天）。正常科研普查通常 \> 30天。 |
| **5** | `data_scope_area` | `params.area_sqkm` | **查询空间范围**。<br>**攻击特征**：同上，攻击者通过已知信息锁定特定海域（如 200 sqkm），而非大范围海域扫描。 |
| **6** | `record_count` | `metrics.input_records` | **处理规模**。<br>用于提供上下文。LSTM 需要知道当前的高 CPU 是消耗在 10000 条数据上（正常）还是 1 条数据上（异常）。 |
| **7** | `export_size_kb` | `metrics.file_size_kb` | **结果文件大小**。<br>**攻击特征**：任务最终导出极小的文件（KB级别），通常是窃取成功的单条轨迹 CSV，而非 MB/GB 级别的统计报告。 |

#### 3\. 序列化与对齐 (Sequence Padding)

由于不同的任务包含的日志条数不同（例如正常任务可能处理 3 个 Batch，攻击任务处理 2 个 Candidate），我们必须统一输入长度以便批量训练。

  * **策略**：Post-Padding（后置填充）。
  * **方法**：找到数据集中最长的任务序列（设为 `Max_L`）。对于长度不足 `Max_L` 的序列，在其尾部填充 `[0, 0, ..., 0]` 向量。
  * **原因**：LSTM 可以通过 Masking 机制忽略填充部分的 0 值，不会影响梯度更新。

#### 4\. 为什么这套方案能检测攻击？

LSTM 不是只看某一个时刻的值，而是看**值的演变**：

1.  **第一阶段**：LSTM 看到 `data_scope_days` 很小（精确查询）。**（怀疑度 +1）**
2.  **第二阶段**：LSTM 看到 `io_ratio` 突然变得极低（漏斗筛选）。**（怀疑度 +2）**
3.  **第三阶段**：LSTM 看到 `record_count` 变为 1，但 `cpu_usage` 和 `duration_ms` 依然很高（针对单个目标的深度挖掘）。**（怀疑度 +5 -\> 判定为攻击）**

这套特征提取方案完美捕捉了轨迹重识别攻击的**时空指纹**。
