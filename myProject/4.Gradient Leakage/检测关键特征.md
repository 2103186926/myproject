# 检测关键特征与JSON日志结构说明

本文档旨在结合代码实现，阐述针对**梯度窃取攻击 (Gradient Leakage)** 的检测特征，并定义用于模型训练的 JSON 日志结构。

## 1. 核心检测特征 (Feature Engineering Plan)

基于《海洋科学云计算任务中的恶意行为检测》文档中 3.4 节 2 的描述，我们在生成日志时植入了以下三个核心维度的特征，这些特征将作为 LSTM 模型的关键输入：

### 1.1 时序特征：异常的计算时长 (Time Duration)

- **特征描述**：恶意节点的本地训练阶段实际上是在执行梯度反演优化，其耗时远超正常的轻量级训练。
- **文档依据**：“恶意客户端的本地计算时间显著且持续地长于其他同类客户端。” 3
- **日志体现**：
  - **正常**：`TRAINING_START` 到 `UPLOAD_SUCCESS` 之间的时间差通常在 **1.5秒** 左右。
  - **恶意**：`TRAINING_START` 到 `UPLOAD_DELAY` 之间的时间差通常在 **20秒以上**，且在日志末尾出现 `Straggler identified` 警告 。

### 1.2 行为特征：优化循环模式 (Optimization Pattern)

- **特征描述**：正常训练进行 Epoch 迭代（通常 < 10），恶意攻击进行 Optimization Iteration（通常 > 100）。
- **文档依据**：“计算热点不在标准的反向传播上，而是在一个需要成千上万次前向传播的优化循环中。” 5
- **日志体现**：
  - **正常**：`phase` 为 `TRAINING_EPOCH`，`metrics.epoch` 计数小。
  - **恶意**：`phase` 为 `ATTACK_ITERATION`，`metrics.iteration` 计数大（如 0/500, 50/500...）。

### 1.3 语义特征：Loss 类型差异 (Loss Type)

- **特征描述**：正常训练旨在最小化预测误差，攻击旨在最小化梯度差异。
- **文档依据**：“Optimization target: Minimizing Gradient Matching Loss.” 6
- **日志体现**：
  - **正常**：`metrics.loss_type` = `prediction_loss`，数值通常较小且平滑下降。
  - **恶意**：`metrics.loss_type` = `gradient_loss`，数值通常较大（如起始 20.0），且伴随 `metrics.tv_loss` (Total Variation Loss) 7。

## 2. JSON 日志格式说明

为了便于后续特征工程提取和 LSTM 处理，我们将非结构化的 Raw Log 转化为以 **Session（一次完整的联邦学习回合）** 为单位的 JSON 对象。

### 2.1 顶层结构 (Session Level)

每个 JSON 对象代表一个客户端在某一轮次中的完整行为序列。

| **字段名**    | **类型** | **说明**                               | **示例**            |
| ------------- | -------- | -------------------------------------- | ------------------- |
| `session_id`  | String   | 全局唯一的会话ID                       | "0"                 |
| `client_id`   | String   | 客户端标识                             | "Node_001"          |
| `label`       | Int      | **标签（Ground Truth）**，用于监督学习 | 0 (正常) / 1 (恶意) |
| `client_type` | String   | 标签的文本描述                         | "malicious"         |
| `logs`        | List     | 该会话内产生的有序日志列表             | `[{...}, {...}]`    |

### 2.2 日志项结构 (Log Entry Level)

`logs` 列表中的每一项代表一行原始日志的结构化信息。

| **字段名**  | **类型** | **说明**                                           | **示例**                                 |
| ----------- | -------- | -------------------------------------------------- | ---------------------------------------- |
| `timestamp` | String   | 精确到毫秒的时间戳，用于计算时序间隔               | "2025-11-27 10:00:05,200"                |
| `level`     | String   | 日志级别，LSTM可将 `WARNING`/`ATTACK` 视为高危特征 | "INFO", "DEBUG", "ATTACK"                |
| `phase`     | String   | **状态机阶段**，这是特征提取的核心键               | `TRAINING_EPOCH`, `ATTACK_ITERATION`     |
| `message`   | String   | 原始日志文本（辅助人工排查）                       | "ITERATION 50/500                        |
| `metrics`   | Object   | 提取出的数值型指标，无需NLP即可直接使用            | `{"loss_value": 18.06, "iteration": 50}` |

## 3. 针对结构化日志的检测方案 (Next Step Strategy)

基于上述结构化数据，我们在下一步（特征提取 + LSTM）中将采取以下策略：

1. **动态时序特征提取**：
   - 计算 `logs` 列表中相邻条目的 **Time Delta (Δt)**。LSTM 将学习恶意节点特有的“长时间无响应”或“密集高频日志”的时间模式。
2. **关键指标序列化**：
   - 提取 `metrics.loss_value` 组成序列。LSTM 极擅长区分“Prediction Loss 下降曲线”和“Gradient Matching Loss 下降曲线”的形态差异。
3. **阶段编码 (Phase Embedding)**：
   - 将 `phase` 字段（如 `TRAINING_EPOCH` vs `ATTACK_ITERATION`）进行 One-Hot 或 Embedding 编码。这是区分攻击行为的最强语义特征。

通过这种结构化模拟，我们不仅保留了文档中描述的所有攻击细节，还消除了非结构化文本解析的噪声，为 LSTM 模型提供了高质量的输入数据。
