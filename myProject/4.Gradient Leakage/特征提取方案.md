# 针对结构化日志的特征提取方案设计

## 1. 方案概述

本方案旨在利用 LSTM (Long Short-Term Memory) 网络处理时间序列数据的能力，检测联邦学习环境中的**梯度窃取攻击 (Gradient Leakage Attack)**。

我们将单个客户端在一次联邦学习回合（Round）中的完整交互日志视为一个**时间序列样本**。不同于传统的基于单条日志规则匹配，本方案通过分析日志序列的**动态演变特征**（如时间间隔变化、状态转移序列、指标趋势），来识别恶意行为模式。

## 2. 特征工程详细设计

基于之前生成的 `federated_learning_structured.json`，我们提取了以下 **16维** 的特征向量用于描述每一个时间步（Log Entry）。

### 2.1 动态时序特征 (Temporal Features)

梯度窃取攻击最显著的物理特征是计算耗时和通信延迟。

- **特征1：时间间隔 (Time Delta, $\Delta t$)**
  - **定义**：当前日志时间戳与上一条日志时间戳的差值（秒）。
  - **检测原理**：
    - **正常行为**：日志生成频率较为稳定，`TRAINING_EPOCH` 之间间隔短且均匀（约0.2s-0.5s）。
    - **恶意行为**：在攻击优化阶段（Iterative Optimization），日志可能会密集产生（如果开了DEBUG）或者长时间静默；最关键的是，在 `UPLOAD` 阶段前会出现巨大的时间断层（Straggler效应，>20s）。
  - **实现**：`curr_time - prev_time`
- **特征2：累计耗时 (Cumulative Time, $T_{cum}$)**
  - **定义**：当前日志距离任务开始（收到全局模型）的总时长。
  - **检测原理**：恶意节点的任务总时长随着攻击迭代次数增加而线性增长，显著偏离正常节点的分布基线。
  - **实现**：`curr_time - start_time`

### 2.2 状态语义特征 (State & Semantic Features)

攻击行为伴随着特定的状态机流转和关键词。

- **特征3-12：阶段独热编码 (Phase One-Hot Encoding)**
  - **定义**：当前日志所属阶段的 One-Hot 向量（10维）。
  - **词表**：`TRAINING_START`, `TRAINING_EPOCH`, `ATTACK_ITERATION`, `FILE_IO`, `UPLOAD_DELAY` 等。
  - **检测原理**：这是最强的语义特征。
    - **正常序列**：`START` -> `EPOCH` -> `EPOCH` ... -> `END` -> `SUCCESS`。
    - **恶意序列**：`START` -> `ATTACK_INIT` -> `ATTACK_ITERATION` ... -> `FILE_IO` -> `UPLOAD_DELAY`。LSTM 能极快地捕捉到 `ATTACK_*` 或 `FILE_IO` 这种异常状态的存在。
- **特征13：日志级别 (Log Level)**
  - **定义**：将日志级别映射为数值 (`DEBUG`:0, `INFO`:1, `WARNING`:2, `ATTACK`:3)。
  - **检测原理**：恶意行为往往伴随着异常的日志级别。例如，未捕获的异常可能产生 `WARNING`，或者安全探针注入的 `ATTACK` 标签。

### 2.3 业务指标特征 (Metric Features)

利用联邦学习特有的数值指标进行区分。

- **特征14：损失值 (Loss Value)**
  - **定义**：直接提取日志中的 `loss_value`。
  - **检测原理**：
    - **正常**：预测损失（Prediction Loss）通常从一个中等值（如 2.x）迅速下降到较小值（0.x）。
    - **恶意**：梯度匹配损失（Gradient Matching Loss）起始值巨大（如 16.0），下降曲线与正常训练截然不同。
- **特征15：损失类型 (Loss Type)**
  - **定义**：将 `loss_type` 映射为数值。`prediction_loss` -> 1, `gradient_loss` -> -1。
  - **检测原理**：通过符号差异，让模型明确区分当前计算的目标函数性质。
- **特征16：计数器 (Iteration Counter)**
  - **定义**：当前的 Epoch数 或 Iteration数，经过归一化处理。
  - **检测原理**：正常训练 Epoch < 10；恶意攻击 Iteration > 100。数量级的差异是判断是否进行“过量计算”的关键依据。

## 3. 下一步：LSTM 模型输入构建

经过 `generate_features.py` 处理后，我们得到的 `lstm_feature_data.json` 将具备以下特性，完美契合 LSTM 的输入要求：

1. **序列化 (Sequential)**：保留了日志的时间先后顺序，使 LSTM 能学习上下文依赖。
2. **定长特征 (Fixed Dimension)**：每个时间步都被压缩为固定的 16 维向量。
3. **变长序列处理**：不同 Session 的日志条数不同（正常约8条，恶意约15条）。在输入模型前，我们将使用 PyTorch 的 `pad_sequence` 进行填充，使它们对齐。

**特征提取逻辑示意图：**

Plaintext

```
[Raw Log Sequence]
  |
  +-> 1. Timestamp Calc  -> [Delta_T, Cum_T]        (捕获 "慢")
  |
  +-> 2. Phase Parsing   -> [0,0,1,0,...]           (捕获 "ATTACK_ITERATION")
  |
  +-> 3. Metric Extract  -> [Loss=16.0, Count=5.0]  (捕获 "异常数值")
  |
  v
[Feature Vector @ t] = [0.2s, 1.5s, 0,0,1..., 16.0, 5.0]
```

这套方案不仅覆盖了文档中提到的“计算时间过长”、“优化循环异常”、“Loss类型不匹配”等核心检测点，还通过结构化数据消除了非结构化日志解析的脆弱性。
