# 梯度窃取检测方案 V3.0：应对隐蔽降频攻击

## 1. 问题回顾与挑战升级

在早期的检测方案中，我们发现模型过于依赖“日志数量”和“总耗时”来区分恶意行为。

然而，在 V3.0 的对抗模型中，我们假设攻击者具备以下反侦察能力：

1. **Log Suppression (日志抑制)**：攻击者在本地运行 500 次迭代，但仅输出 5 条日志，使得其日志序列长度与正常节点一致。
2. **Time Mimicry (时间伪装)**：攻击者利用高性能硬件加速，使得跑 500 次迭代的总耗时与正常节点跑 5 次 Epoch 的耗时接近。

这意味着：单纯看 Log Count 和 Duration 已经失效。

## 2. 破局思路：微观算力指纹 (Micro-Computational Fingerprinting)

尽管攻击者可以伪装宏观指标（总时间、总条数），但受限于物理规律，他们无法掩盖**“单位计算密度”**。

- **正常节点**：`5个 Epoch / 5秒` -> `1 Epoch / 1秒` (慢速计算)
- **恶意节点**：`500个 Iteration / 5秒` -> `100 Iterations / 1秒` (极速计算)

如果攻击者为了掩盖日志数量，每 100 个 Iteration 才打印一条日志，那么我们就会观察到：**日志之间的时间间隔($\Delta t$)虽然正常，但内部包含的逻辑步数($\Delta Step$)却异常巨大。**

## 3. V3 特征工程详解

我们从每条日志中提取 **7维** 特征，旨在捕捉这种算力密度的差异。

| **特征维度** | **名称**                      | **提取逻辑**                   | **对抗防御原理**                                             |
| ------------ | ----------------------------- | ------------------------------ | ------------------------------------------------------------ |
| **Dim 1**    | **Delta Time ($\Delta t$)**   | $t_{current} - t_{prev}$       | 宏观时间流逝。在 V3 中，黑白样本的 $\Delta t$ 可能很接近（都是 1s 左右），不再是主要区分特征，但用于计算衍生特征。 |
| **Dim 2**    | **Step Gap (步长跳跃)**       | $Step_{current} - Step_{prev}$ | **核心特征 (Kill Chain)**。正常训练是连续的 (Gap=1)；降频攻击必然导致跳跃 (Gap=100)。攻击者无法掩盖这一点，除非他真的只跑 5 步（那样攻击就失败了）。 |
| **Dim 3**    | **Time per Index (单步算力)** | $\Delta t / StepGap$           | **核心特征**。表示“完成一个逻辑步需要多久”。恶意节点的数值（如 0.01s）会远小于正常节点（如 1.0s）。这是物理算力的直接体现。 |
| **Dim 4**    | **Loss Value**                | Raw Value                      | 梯度匹配 Loss 的绝对数值通常远大于正常预测 Loss（如 20.0 vs 2.0），即使归一化后，其分布区间也不同。 |
| **Dim 5**    | **Loss Velocity**             | $\Delta Loss / \Delta t$       | 攻击初期的 Loss 下降极其剧烈（震荡），而正常训练较为平滑。   |
| **Dim 6**    | **Is Compute Phase**          | One-Hot                        | 标识当前是否在计算阶段，辅助模型聚焦于计算阶段的特征，忽略 Upload 阶段的噪音。 |
| **Dim 7**    | **Cumulative Time Norm**      | $T_{cum} / 30s$                | 辅助 LSTM 感知任务进度的相对位置。                           |

## 4. 预期效果

通过这套方案，我们将在 CSV 可视化中看到：

- **正常样本**：
  - `step_gap` 列基本全为 **1.0**。
  - `time_per_index` 列数值较大（如 **0.5 - 1.5**）。
- **恶意样本（即使是隐蔽的）**：
  - `step_gap` 列会出现 **50.0, 100.0** 这样的尖峰。
  - `time_per_index` 列数值极小（如 **0.005**）。

LSTM 模型将极易捕捉到 `Step Gap` 和 `Time per Index` 的序列异常模式，从而在攻击者试图通过“少打日志”来伪装时，反而暴露了其“计算密度过高”的马脚。
