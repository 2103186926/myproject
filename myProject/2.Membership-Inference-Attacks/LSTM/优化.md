### 1. 如何提取时序关系并存入JSON特征文件
为了让LSTM模型能够理解用户任务的先后顺序，我们必须打破原先的“一任务一行（CSV）”的静态聚合模式，转向“一任务一序列（JSON）”的动态结构。
(A) 时序关系提取流程
    分组 (Grouping)：对所有日志条目（uploaded:日志样例.json中的事件）按其唯一的任务ID (context.taskId) 进行聚合。
    排序 (Ordering)：在每个任务组内，严格按照高精度时间戳 (timestamp) 对所有事件进行升序排序。
    筛选 (Filtering)：只提取与模型查询相关的事件 (eventType: "model_query")，形成一个有序的查询行为序列。其他如 task_start 或 internal_activity 事件中的信息则作为“全局特征”提取。

#### 模板格式定义：
{
  "task_def456": { // 任务ID
    "is_malicious": 1, // 任务标签 (0/1)
    "global_features": {
      "attack_model_trained": true,       // 全局特征1: 是否训练了攻击模型
      "total_query_count": 151            // 全局特征2: 总查询次数（用于资源失衡分析）
    },
    "sequence": [  // 时序特征序列
      {
        "timestep": 0,  // 时间步长，从0开始递增
        "response_confidence": 0.9876,    // 当前查询事件的置信度
        "time_delta_since_last_query": 0.0,  // 时间间隔（与上一个查询事件的时间差）
        "distance_from_anchor_vector": 0.0 // 此刻锚点被确定为 0.9876 的样本
      },
      {
        "timestep": 1,
        "response_confidence": 0.4521,
        "time_delta_since_last_query": 0.205, // 时间间隔
        "distance_from_anchor_vector": 1.85   // 距离锚点较远（影子样本）
      },
      // ... 更多 timesteps ...
    ]
  },
  // ... 更多任务 ...
}

新的特征向量文件被设计为两部分：sequence（时序特征）和 global_features（全局特征）。
(A) sequence (时序特征)
这些特征在序列的每一步（即每一次model_query事件） 都会被计算一次。
    response_confidence 浮点数 (0-1):
        含义: 当前model_query事件的details.queryResponse.confidence值。
        目的: 捕捉“置信度分布异常”。恶意序列会呈现 [高, 低, 低, ...] 的模式。

    time_delta_since_last_query 浮点数 (秒):
        含义: 当前model_query事件的timestamp与上一个model_query事件timestamp之间的时间差（秒）。
        目的: 捕捉“持续查询”。恶意序列的时间差会非常小且有规律（例如，都接近0.2秒）。

    distance_from_anchor_vector 浮点数(欧氏距离):
        含义: 当前查询的输入向量 (queryInput.featureVector) 与该任务置信度最高的锚点向量之间的欧氏距离。
        目的: 捕捉高密度查询聚类 (恶意任务的该值会持续为极小值，因查询都围绕敏感样本)。恶意序列的该值会是 [0, 极小值, 极小值, ...]。

(B) global_features (全局特征)
这些特征是对整个任务的总结，在序列中只出现一次，用于辅助LSTM判断。
    attack_model_trained 布尔值 (0/1):
        含义: 在整个任务日志中，是否出现过eventType为internal_activity且activityType为attack_model_training的事件。
        目的: 捕捉“寄生模型训练”。寄生模型训练（意图的决定性证据）。

    final_output_size_bytes (整数):
        含义: task_end事件中记录的details.finalOutputSizeBytes值。任务生命周期内对模型发起的总查询次数。
        目的: 捕捉“微小输出”。持续查询（资源滥用）。可与最终输出大小结合分析失衡。



### 3. 基于LSTM时序特征提取的详细日志检测方案（修正版）

此方案的核心在于将用户的行为序列（时序特征）和任务的全局意图（全局特征）结合起来，通过一个混合神经网络进行分类。

步骤 1: 任务日志预处理与序列化

    数据采集：实时或批量从结构化日志流中采集数据。

    序列构建：按 context.taskId 分组，按 timestamp 排序，提取所有 model_query 事件，构建原始序列。

    确定“锚点向量” (Anchor Vector Determination)：

        遍历每个任务的序列，找到 queryResponse.confidence 最高的查询 Qmax_conf​。

        将该查询的输入特征向量 Qmax_conf​.queryInput.featureVector 定义为该任务的锚点向量 Vanchor​。

    特征计算：遍历序列中的每个查询 Qi​，计算新的时序特征：

        Qi​.response_confidence

        Qi​.time_delta_since_last_query

        Qi​.distance_from_anchor_vector=EuclideanDistance(Qi​.queryInput.featureVector,Vanchor​)

步骤 2: 建立LSTM混合输入模型

为了最大化特征利用率，模型采用双路径混合架构：

    时序输入 (Xt​):

        输入层：接收三维时序特征（response_confidence, time_delta_since_last_query, distance_from_anchor_vector）。

        填充 (Padding)：由于序列长度不等，需要对所有任务序列进行零填充 (Zero-Padding) 或截断 (Truncating)，使输入张量的形状统一为 (批次大小, 最大序列长度 $T$, 3)。

        LSTM/GRU 层：核心处理层。使用门控循环单元（GRU）或长短期记忆网络（LSTM）来学习特征之间的依赖关系和时间模式。

        序列总结：从LSTM层提取最终隐藏状态 hT​，它是一个固定长度的向量，代表了整个任务序列的时序特征总结。

    全局输入 (Xg​):

        输入层：接收二维全局特征（attack_model_trained, total_query_count）。

        处理层：通常通过一个简单的全连接（Dense）层进行维度对齐和初步特征抽象。

    特征融合与分类：

        融合 (Concatenation)：将 hT​（序列总结）和处理后的 Xg​（全局意图）向量在特征维度上拼接起来。

        分类器：将拼接后的向量输入到一个或多个全连接（Dense）层，最后通过一个 Sigmoid 激活函数，输出任务为恶意 (P(Malicious)) 的概率。

步骤 3: 模型训练与评估

    损失函数：对于二分类任务，使用二元交叉熵 (Binary Cross-Entropy) 作为损失函数。

    分组交叉验证 (GroupKFold)：至关重要。在评估模型性能时，必须使用任务ID (taskId) 作为分组依据进行 K 折交叉验证。这确保了同一任务的所有时间步都位于同一个训练集或测试集中，从而防止数据泄露，并获得对模型泛化能力的真实评估。

    性能指标：除了准确率（Accuracy），应重点关注召回率 (Recall) 和 F1-Score，因为检测系统需要最大限度地找出真正的攻击行为（即高召回率），同时保持较低的误报率。
