# 针对隐蔽攻击的高混淆度检测方案 (V3 - Research Grade)

## 1. 设计哲学：为什么之前的方案失败了？

之前的方案虽然达到了 100% 的准确率，但在科研语境下是失败的。因为它依赖了“强特征”（如 `Is_Training`），这使得问题退化为简单的规则匹配（If-Else），失去了使用深度学习（LSTM）挖掘时序模式的必要性。

**V3 方案的核心目标**：

- **移除上帝视角**：假设防御者只能看到 API 网关日志（仅包含时间戳、请求参数），看不到客户端内部行为（如是否在训练模型、文件保存成什么后缀）。
- **制造混淆**：引入“合法高频脚本”作为干扰项，使恶意行为与正常行为在统计分布上产生重叠（Overlap）。
- **增加噪声**：模拟真实环境的不确定性，迫使模型学习鲁棒的特征表示。

## 2. 数据集构成：三足鼎立

为了构建非线性的决策边界，我们设计了三类用户，制造“真假难辨”的局面：

| 用户类型           | 行为特征                             | 统计学表现 (Mean/Std)    | 标签          |
| ------------------ | ------------------------------------ | ------------------------ | ------------- |
| **Normal Human**   | 极低频，长间隔思考，随机性强         | Mean间隔大，Std极大      | 0 (Normal)    |
| **Legit Script**   | **高频**，极度规律，定时任务         | Mean间隔小，**Std接近0** | 0 (Normal)    |
| **Smart Attacker** | **中高频**，伪造的随机性，间歇性休息 | Mean间隔中，**Std中等**  | 1 (Malicious) |

**难点在于**：LSTM 必须学会区分“极度规律的机器脚本”和“伪造随机的攻击脚本”，这比区分“人 vs 机器”要难得多。

## 3. 特征工程详解：从“量”到“质”

我们提取 7 维特征，不再依赖单一的绝对值，而是重点关注**滑动窗口下的二阶统计量**。

### 3.1 特征维度说明

| 索引 | 特征名称        | 物理含义              | 提取方式                               | 对抗原理 (为何能检测?)                                       |
| ---- | --------------- | --------------------- | -------------------------------------- | ------------------------------------------------------------ |
| 0    | **Time_Delta**  | **瞬时时间间隔**      | $t_i - t_{i-1}$                        | 基础特征。人类操作通常 $>5s$，脚本通常 $<1s$。攻击者会试图模仿人类但保持效率。 |
| 1    | **Q_Count**     | **单次请求量**        | 直接读取日志 `details.count`           | 基础特征。攻击者可能为了掩人耳目将大批量请求拆分为小批量，混淆视听。 |
| 2    | **Cum_Q**       | **累计查询量**        | $\sum_{k=0}^{i} Q\_Count_k$            | **状态特征**。随着时间推移，即使攻击者单次查得少，其累计窃取量也会显著高于正常人类（Throughput accumulation）。 |
| 3    | **Delta_Std**   | **间隔标准差** (核心) | 最近 $N$ 次 `Time_Delta` 的标准差      | **破局关键**。合法脚本极其规律 ($Std \approx 0$)；人类极不规律 ($Std \gg 10$)；**聪明攻击者为了伪装随机性，Std 会呈现出一种“不自然的中等波动”**。 |
| 4    | **Delta_Mean**  | **平均间隔**          | 最近 $N$ 次 `Time_Delta` 的均值        | 反映当前的交互频率基线。                                     |
| 5    | **Throughput**  | **平均吞吐量**        | $Cum\_Q / (t_i - t_{start})$           | 长期来看，攻击者的资源消耗效率通常高于人类，低于暴力脚本。   |
| 6    | **Burst_Ratio** | **突发比率**          | $Q\_Count_i / Mean(Q\_Count_{window})$ | 检测异常的流量激增。攻击者可能在深夜或某些时刻突然加速以追赶进度，产生突发信号。 |

### 3.2 特征提取实现细节

1. **滑动窗口 (Sliding Window)**：
   - 设置窗口大小 $W=5$。
   - 维护一个队列 `history_deltas` 和 `history_counts`。
   - 每处理一条日志，计算窗口内的 `np.std()` 和 `np.mean()`。
2. **噪声注入 (Noise Injection)**：
   - 为了模拟真实环境（网络抖动、系统延迟）并防止模型过拟合特定数值，我们在特征生成阶段人为加入高斯噪声。
   - 公式：$Feature_{final} = Feature_{raw} + \epsilon, \quad \epsilon \sim N(0, 0.05)$
3. **归一化处理**：
   - 在训练前，所有特征将通过 Z-Score标准化 ($x' = \frac{x - \mu}{\sigma}$)，确保 LSTM 收敛。

## 4. 预期实验结果

- **训练集 Loss**：下降曲线应带有震荡，不再是平滑直线下坠，说明模型在努力寻找决策边界。
- **测试集表现**：
  - **Human**: 100% 识别准确率（特征差异巨大）。
  - **Legit Script vs Smart Attacker**: 预计会有 **5% - 10% 的混淆区**。
  - **最终 Accuracy**: 预期落在 **92% - 96%** 区间。

## 5. 方案优势（论文卖点）

1. **无上帝视角**：仅依赖 API 网关最基础的日志数据，无需侵入客户端，具备极高的落地可行性。
2. **抗干扰性强**：通过引入“合法高频脚本”作为负样本，证明了模型不仅仅是简单的阈值检测，而是真正学习到了“攻击者的伪装模式”。
3. **鲁棒性**：噪声注入机制保证了模型在网络波动环境下的稳定性。
