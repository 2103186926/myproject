# 针对结构化日志的模型窃取检测方案

## 1. 方案概述

本方案旨在将结构化的 JSON 日志数据转化为适用于深度学习（如 LSTM/GRU）的时序特征矩阵。不同于传统的基于统计（如 QPS 阈值）的检测方法，本方案利用**动态时序特征**来捕捉攻击者的行为模式（Contextual Behavior Patterns）。

**核心思想**：模型窃取攻击不仅仅是“查询量大”，更是一个包含“探测-批量查询-训练-保存”的特定动作序列。

## 2. 特征提取详细设计 (Feature Engineering)

我们为每个时间步（Log Entry）设计了一个 **6维** 的特征向量。

### 2.1 维度定义

| 索引  | 特征名称            | 数据类型          | 物理含义与检测逻辑                                           |
| ----- | ------------------- | ----------------- | ------------------------------------------------------------ |
| **0** | **Event_ID**        | Categorical (Int) | **事件类型编码**。将 `model_training` (ID:5) 与 `model_inference` (ID:4) 区分开是识别攻击意图的关键。正常用户极少在推理阶段进行训练。 |
| **1** | **Time_Delta**      | Numerical (Float) | **时间间隔**（秒）。攻击脚本通常具有机器级的高频操作特征（间隔极短且稳定），而人类操作会有较长且随机的思考时间。 |
| **2** | **Query_Count**     | Numerical (Int)   | **批量查询大小**。攻击者为了高效窃取，倾向于使用大 Batch (如 1000+)，而正常单点查询通常为 1-10。这是强度的体现。 |
| **3** | **Is_Training**     | Binary (0/1)      | **训练行为标记**。这是最为“致命”的特征。如果在日志流中检测到此位为 1，且伴随着高查询量，基本可判定为恶意行为。 |
| **4** | **R2_Score**        | Numerical (Float) | **替代模型性能**。如果此数值随时间步推移呈现上升趋势（如 0.4 -> 0.6 -> 0.8），说明攻击者正在逐步逼近目标模型。 |
| **5** | **Suspicious_File** | Binary (0/1)      | **敏感文件产出**。标记是否生成了 `.joblib` 或 `.pkl` 等模型文件。正常任务通常产出 `.pdf` 或 `.png`。 |

### 2.2 序列处理 (Sequence Handling)

由于不同会话的日志长度不一（正常任务可能只有 6 条日志，攻击任务可能有 20+ 条），我们需要统一输入维度。

- **最大长度 (Max Seq Len)**：扫描整个数据集，取最长会话的长度 $L_{max}$。
- **填充 (Padding)**：对于长度小于 $L_{max}$ 的会话，在其**尾部**填充全 0 向量。
  - *注*：LSTM 模型可以通过 Masking 机制忽略这些填充位，或者学习到全 0 代表任务结束。

### 2.3 数据形态变换

原始 JSON 数据经过处理后，将转化为以下形态的 3D 张量，直接输入 PyTorch/TensorFlow 模型：

$$\text{Input Tensor Shape} = (N, L_{max}, D)$$

- $N$: 样本数量（Session Count，例如 120 个会话）
- $L_{max}$: 时间步长（例如 25 步）
- $D$: 特征维度（6 维）

## 3. 为什么这样设计？（结合业务场景）

### 场景 A：正常科研任务

- **特征流**：
  1. `Event: Data_Load` -> `Query_Count: 0`
  2. `Event: Inference` -> `Query_Count: 5` (量小)
  3. `Event: Visualization`
  4. `Event: Save` -> `Suspicious: 0` (.pdf)
- **LSTM 视角**：这是一个短序列，没有训练行为，查询量低，最终输出安全文件。-> **判定：Normal**

### 场景 B：模型窃取攻击

- **特征流**：
  1. `Event: Input_Gen` (特殊事件)
  2. `Event: Inference` -> `Query_Count: 1000` (高频)
  3. `Event: Inference` -> `Query_Count: 1000`
  4. `Event: Training` -> `Is_Training: 1`, `R2: 0.45` (异常点)
  5. `Event: Inference` ...
  6. `Event: Save` -> `Suspicious: 1` (.joblib)
- **LSTM 视角**：捕捉到了“高强度查询”紧接着“训练行为”的时序依赖，且 R2 分数在动态变化。-> **判定：Malicious**

## 4. 下一步建议 (Step 3 Preview)

在生成了 `lstm_feature_data.json` 后，下一步构建 LSTM 模型时，建议：

1. 使用 **Embedding 层** 来处理 `Event_ID` 特征（虽然只有 9 类，但 Embedding 能更好地表示语义距离）。
2. 对 `Query_Count` 和 `Time_Delta` 进行 **StandardScaler (Z-Score)** 标准化，防止数值过大影响梯度下降。
