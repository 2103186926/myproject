### 问题分析与解决方案

**问题 1：特征值过于稀疏（很多 0）**

- **原因**：目前的特征提取是“**无状态的（Stateless）**”。每条日志只触发特定的特征。例如，`data_load` 事件只有 Event_ID，而 `Query_Count`、`R2_Score` 全是 0。这导致矩阵中绝大多数位置都是空的。
- **解决方案**：引入“**有状态（Stateful）/ 累计（Cumulative）**”特征。
  - 与其只记录“当前这一步查了多少次”，不如记录“**这也是本会话累计第多少次查询**”。这样，随着时间推移，该特征列会持续有值且不断增大，不再是 0。
  - 对于 `R2_Score`，采用“**采样保持（Sample-and-Hold）**”策略。如果这一步没训练，就保持上一步的分数，而不是归零。

**问题 2：Normal 和 Malicious 太容易区分（攻击力度太明显）**

- **原因**：正常用户每次查 1-5 个，攻击者每次查 1000 个；正常用户不做训练，攻击者频繁训练。这两个分布完全没有重叠（Overlap）。
- **解决方案**：实施“**隐蔽攻击（Stealthy Attack）**”与“**专家用户（Power User）**”策略。
  - **削弱攻击者**：攻击者不再每次查 1000 个，而是把大批次拆分成小的“微批次”（例如每次 20-50 个），混入正常流量中。
  - **增强正常用户**：模拟“专家用户”或“自动化测试脚本”，它们也会进行较大批量的查询（例如验证数据集），造成特征重叠。
  - **增加噪声**：攻击者会在查询中间穿插无意义的 `data_processing`（伪装成正在分析数据），打断连续查询的序列特征。



# 针对隐蔽模型窃取攻击的特征工程方案 (V2)

## 1. 方案背景更新

针对 V1 版本存在的“特征稀疏”和“区分度过高”问题，V2 方案进行了以下核心升级：

1. **引入状态机 (State Machine)**：不再孤立看待每一条日志，而是提取“截至当前时间步”的**累积状态**，解决稀疏性问题。
2. **重叠分布设计**：针对隐蔽攻击者（Stealthy Attacker）和专家用户（Power User）行为高度相似的情况，设计更细粒度的动态特征。

## 2. 特征维度详解 (8维向量)

现在的输入向量从 6 维增加到 8 维，且富含稠密信息。

| 索引 | 特征名          | 类型 | 状态属性           | 物理含义与对抗逻辑                                           |
| ---- | --------------- | ---- | ------------------ | ------------------------------------------------------------ |
| 0    | **Event_ID**    | Cat  | 瞬时               | 事件类型的数值编码。                                         |
| 1    | **Time_Delta**  | Num  | 瞬时               | 距离上一条日志的时间。**对抗点**：隐蔽攻击者会故意随机化这个值，LSTM 需要学习其分布的方差（Variance）。 |
| 2    | **Curr_Batch**  | Num  | 瞬时               | 当前这一步请求的查询量。**难点**：攻击者现在的 Batch 只有 30-50，与 Power User 类似。单独看此特征已无法区分。 |
| 3    | **Cum_Queries** | Num  | **累积(Stateful)** | **核心稠密特征**。本会话累计查询总量。随着时间推移，攻击者的总量往往会超过 Power User（尽管单次不高，但次数多）。 |
| 4    | **Is_Train**    | Bin  | 瞬时               | 是否为训练事件。**对抗点**：攻击者减少了训练频率，此特征变得更稀疏，但在序列中只要出现一次就是强信号。 |
| 5    | **Last_R2**     | Num  | **保持(Hold)**     | **核心稠密特征**。记录最近一次观测到的模型分数。如果在 Inference 阶段此值一直保持且较高（如 >0.6），说明该用户持有一个经过训练的模型。 |
| 6    | **Intensity**   | Num  | **衍生(Derived)**  | `Cum_Queries / Elapsed_Time`。查询强度。即使攻击者加了 sleep，长期运行下来，其平均强度可能仍异于常人。 |
| 7    | **Susp_File**   | Bin  | 瞬时               | 是否保存了可疑后缀文件（包含 `.dat`, `.bin` 等模糊后缀）。   |

## 3. 为什么这样能解决问题？

### 3.1 解决稀疏性 (Sparsity)

- **旧方案**：`[2, 0.5, 0, 0, 0, 0]` (Data Load 事件，后面全是 0)
- **新方案**：`[2, 0.5, 0, 1500, 0, 0.75, 5.2, 0]`
  - 即使是 Data Load 事件，我们也能看到 `Cum_Queries=1500`（之前查了很多）和 `Last_R2=0.75`（之前训练过）。
  - **结果**：特征矩阵不再主要是 0，LSTM 每个时间步都能获得上下文信息。

### 3.2 解决区分度 (Difficulty)

- **Power User**: 单次 Batch=50，Cum_Queries=500，Intensity=高，Last_R2=0。
- **Stealthy Attacker**: 单次 Batch=30，Cum_Queries=1200，Intensity=中，Last_R2=0.6。
- **LSTM 的任务**：不再是简单的 `if Batch > 500 then Malicious`，而是要学会：
  - “虽然单次查询不多，但累计量一直在涨，而且中间夹杂了 R2 分数的变动，这是一个长周期的窃取行为。”
  - 这种非线性的时序模式正是 LSTM 擅长捕捉的。

## 4. 后续模型建议

由于数据变复杂了（Normal 和 Malicious 有特征重叠），建议 LSTM 模型结构如下：

- **Input**: (Batch, Max_Len, 8)
- **Layers**: 2层 LSTM (Hidden Size 64) + Dropout (0.3)
- **Head**: Fully Connected -> Sigmoid
- **Loss**: Binary Cross Entropy
