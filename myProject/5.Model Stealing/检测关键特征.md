# 模型窃取攻击检测：关键特征与日志结构说明

本文档旨在定义用于检测海洋科学云平台中“模型窃取攻击”的关键特征，并详细说明 JSON 日志的结构。这些定义将直接指导后续的特征提取（Feature Extraction）和 LSTM 模型输入向量的构建。

# 关键检测特征 (Key Features)

基于之前的业务分析，我们将从 JSON 日志中提取以下三类特征。LSTM 模型将通过学习这些特征随时间的变化序列来进行分类。

### 2.1 频率与强度特征 (Frequency & Intensity)

攻击者通常需要大量数据来训练替代模型，因此查询频率和批量大小是首要特征。

- **特征点**：`details.query_count`
- **恶意模式**：单次请求的查询量巨大（如 `1000`），且在短时间内连续出现多次。
- **正常模式**：查询量通常较小（如 `1-10`），且是偶发的。
- **特征工程逻辑**：
  - 提取每条日志的 `query_count`，若无则为 0。
  - 序列中会出现持续的高数值峰值。

### 2.2 行为序列特征 (Behavioral Sequence)

攻击者的操作流程与正常科研任务有本质区别。

- **特征点**：`event_type` 的转换序列
- **恶意模式**：
  1. `input_generation` (生成对抗样本/均匀分布样本)
  2. `model_inference` (循环多次)
  3. `model_training` (训练替代模型 - **强恶意信号**)
  4. `file_save` (保存模型)
- **正常模式**：
  1. `data_load`
  2. `model_inference` (一次)
  3. `data_processing` (可视化)
  4. `file_save` (保存报告/图片)
- **特征工程逻辑**：对 `event_type` 进行 One-Hot 编码或 Label Encoding。例如：`model_training` 出现意味着极高风险。

### 2.3 结果与产物特征 (Artifacts & Metrics)

攻击的目标是获得一个性能良好的替代模型。

- **特征点 A**：`details.r2_score` (模型性能)
  - **恶意模式**：该字段存在，且随时间推移呈**上升趋势** (0.3 -> 0.6 -> 0.9)。
  - **正常模式**：通常不存在该字段，或者该字段用于评估静态数据的拟合度，不会在任务中动态提升。
- **特征点 B**：`details.file_ext` (文件后缀)
  - **恶意模式**：`.joblib`, `.pkl`, `.pth` (直接保存了可执行的模型对象)。
  - **正常模式**：`.png`, `.pdf`, `.nc` (保存的是数据结果或图表)。

## 3. 针对结构化日志的检测方案 (特征提取准备)

为了输入 LSTM，我们需要将上述 JSON 对象的列表转化为固定维度的**数值矩阵**。

### 3.1 步骤规划 (Step 2 Preview)

对于每个 Session 的 `logs` 列表，我们将提取一个形状为 `(Sequence_Length, Feature_Dim)` 的矩阵。

### 3.2 特征向量定义

建议提取以下维度作为 LSTM 的输入特征（Feature Dim）：

1. **Event_Type_ID**: (Category) 事件类型的索引 (0: start, 1: inference, 2: training, 3: save...)
2. **Query_Volume**: (Numerical) `details.query_count` 的归一化数值。
3. **Is_Training**: (Binary) 当前日志是否包含训练行为 (`event_type == 'model_training'`)。
4. **Model_Performance**: (Numerical) `details.r2_score`，若无则为 -1 或 0。
5. **Is_Suspicious_File**: (Binary) `details.file_ext` 是否为模型后缀 (`.joblib`/`.pkl`)。
6. **Time_Delta**: (Numerical) 距离上一条日志的时间间隔（检测高频自动化脚本）。

**示例向量转换：**

| 原始日志 (Simplified)    | Event_ID | Query_Vol | Is_Training | R2_Score | Suspicious_File |
| ------------------------ | -------- | --------- | ----------- | -------- | --------------- |
| `task_start`             | 0        | 0         | 0           | 0        | 0               |
| `input_gen`              | 1        | 0         | 0           | 0        | 0               |
| `inference (count=1000)` | 2        | 1000      | 0           | 0        | 0               |
| `inference (count=1000)` | 2        | 1000      | 0           | 0        | 0               |
| `training (r2=0.5)`      | 3        | 0         | 1           | 0.5      | 0               |
| `file_save (.joblib)`    | 4        | 0         | 0           | 0.9      | 1               |

通过这种转换，LSTM 将能够捕捉到“**大量查询后接模型训练，且 R2 分数上升**”这一独特的时间序列模式。



# JSON 日志格式说明 (Schema)

为了便于机器学习模型处理，我们将原始日志标准化为 JSON 格式。数据以“会话 (Session/Task)”为单位进行组织。

### 数据层级结构

```
[
  {
    "session_id": "task_a1b2c3d4",
    "label": "malicious",  // 或 "normal"
    "logs": [ ... ]        // 该任务产生的时间序列日志列表
  },
  ...
]
```

每条日志包含以下字段，其中 `details` 字段是特征提取的核心来源。

| 字段名       | 类型   | 说明                                           | 示例                                      |
| ------------ | ------ | ---------------------------------------------- | ----------------------------------------- |
| `timestamp`  | String | ISO 8601 时间戳                                | `2023-10-27T10:00:00`                     |
| `event_type` | String | **核心字段**：事件类别，决定了行为语义         | `model_inference`, `file_save`            |
| `message`    | String | 原始日志文本                                   | `Querying target model batch`             |
| `details`    | Dict   | **核心字段**：结构化的元数据，用于提取数值特征 | `{"query_count": 1000, "r2_score": 0.85}` |

`simulated_logs.json` 文件是一个 JSON 数组（List），其中每一个元素代表一个完整的**用户会话（Session）或计算任务（Task）**。

## 1. 第一层：会话对象 (Session Object)

这是数据的最外层单元。所有的分析都是以“会话”为单位进行的，LSTM 模型也是对一个会话中的日志序列进行分类。

```
{
  "session_id": "task_26356075",
  "label": "normal",
  "logs": [ ... ]
}
```

| Key              | Value 类型 | 含义说明                                                     | 示例值                                |
| ---------------- | ---------- | ------------------------------------------------------------ | ------------------------------------- |
| **`session_id`** | String     | 任务的唯一标识符。在特征提取时用于区分不同的样本。           | `"task_26356075"`                     |
| **`label`**      | String     | **Ground Truth 标签**。标记该任务是正常科研行为还是恶意攻击。我们在训练模型时用它作为目标值（y）。 | `"normal"` (正常)`"malicious"` (恶意) |
| **`logs`**       | Array      | **日志序列**。包含该任务从开始到结束产生的所有日志条目。这就是我们要输入给 LSTM 的时序数据。 | `[{...}, {...}]`                      |

## 2. 第二层：日志条目对象 (Log Entry Object)

`logs` 数组中的每一个对象代表一条具体的日志记录。这相当于传统文本日志中的“一行”。

```
{
  "timestamp": "2025-12-01T17:38:34.183948",
  "task_id": "task_26356075",
  "user_id": "user_1933",
  "level": "INFO",
  "event_type": "data_load",
  "message": "Loading dataset from storage",
  "details": { ... }
}
```

| Key              | Value 类型 | 含义说明                                                     | 关键点                                                       |
| ---------------- | ---------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **`timestamp`**  | String     | ISO 8601 格式的时间戳。                                      | 用于计算**`Time_Delta` (时间间隔)** 特征，判断操作频率。     |
| `task_id`        | String     | 关联的 session_id，用于数据完整性校验。                      | -                                                            |
| `user_id`        | String     | 操作用户的 ID。                                              | 在多用户分析中很有用（如发现同一用户发起了多个恶意任务）。   |
| `level`          | String     | 日志级别 (INFO, WARN, ERROR)。                               | 本次模拟主要为 INFO。                                        |
| **`event_type`** | String     | **核心字段！事件类型**。它定义了当前操作的语义（是查询？是训练？还是保存文件？）。 | 直接映射为特征向量中的 **`Event_ID`**。                      |
| `message`        | String     | 原始的文本日志消息。                                         | 给人类阅读的描述，特征提取时主要依赖 `details` 和 `event_type`。 |
| **`details`**    | Object     | **核心字段！详细参数**。这是一个字典，包含了从非结构化文本中提取出来的数值和属性。 | 不同的 `event_type` 对应不同的 `details` 内容（详见下文）。  |

## 3. 第三层：Details 对象详解 (Context-Aware)

`details` 字段的内容是动态变化的，它取决于 `event_type`。这是我们提取高维特征（如查询量、R2分数）的主要来源。

### A. 正常行为常见事件

#### 1. `data_load` (数据加载)

- **含义**: 科研人员加载 `.nc` (NetCDF) 等气象数据文件。
- **Details**:
  - `file_type`: 文件类型 (e.g., "nc")
  - `size_mb`: 文件大小

#### 2. `data_processing` (数据处理/可视化)

- **含义**: 生成热力图、流场图等。
- **Details**:
  - `output_type`: 输出类型 (e.g., "image")

### B. 恶意行为关键事件 (重点关注)

#### 1. `input_generation` (输入生成)

- **含义**: 攻击者生成用于探测模型的查询向量（通常使用自适应策略）。
- **Details**:
  - `strategy`: 生成策略 (e.g., `"adaptive"`, `"random"`)。
  - `count`: 生成的数量 (e.g., `24721`)。**正常任务极少有此步骤。**

#### 2. `model_inference` (模型推理/查询)

这是区分正常与恶意的最关键环节。

- **恶意样本**:

  ```
  "details": {
    "model": "wave_forecast_v3",
    "query_count": 1000,      // <--- 异常高：单次请求查1000个点
    "strategy": "batch",      // <--- 策略：批量
    "total_stolen": 2000      // <--- 累计窃取量
  }
  ```

- **正常样本**:

  ```
  "details": {
    "model": "wave_forecast_v3",
    "query_count": 2,         // <--- 正常低：单次请求查2个点
    "strategy": "manual"      // <--- 策略：手动/少量
  }
  ```

- **特征提取**: 我们会直接提取 `query_count` 的值作为特征。

#### 3. `model_training` (替代模型训练)

这是**实锤证据**。正常用户调用平台API是为了获取预测结果，绝不会在平台日志里表现出“我在训练一个随机森林”。

- **Details**:
  - `model_type`: 训练的模型类型 (e.g., `"RandomForest"`).
  - `r2_score`: **模型性能评分** (e.g., `0.7723`).
- **特征提取**: 提取 `r2_score`。如果在时间序列中发现该值存在且逐渐升高，LSTM 会捕捉到这个“学习过程”。

#### 4. `file_save` (文件保存)

- **恶意样本**: 保存模型文件或数据集。
  - `filename`: `"stolen_model.joblib"`, `"stolen_data.csv"`
  - `file_ext`: `".joblib"`, `".csv"` (结合上下文)
- **正常样本**: 保存报告。
  - `filename`: `"report_2023.pdf"`
  - `file_ext`: `".pdf"`
- **特征提取**: 判断 `file_ext` 是否在黑名单（如 `.joblib`, `.pkl`）中，生成 `Suspicious_File` (0/1) 特征。

## 4. 总结：数据流向

从原始 JSON 到 LSTM 输入的转化逻辑如下：

1. **读取 JSON**: 获取 `logs` 列表。
2. **遍历 Logs**:
   - 看到 `event_type="model_inference"` -> 提取 `details.query_count` (例如 1000)。
   - 看到 `event_type="model_training"` -> 标记 `Is_Training=1`，提取 `details.r2_score`。
   - 计算当前日志与上一条日志的 `timestamp` 差值 -> `Time_Delta`。
3. **生成向量**: 将上述提取的值组合成一个向量 `[Event_ID, Time_Delta, Query_Count, ...]`。
4. **形成序列**: 一个 Session 的所有向量组成一个矩阵，作为 LSTM 的一个样本输入。
