# 高级时空关联攻击检测：新版特征工程方案综述

## 1. 设计哲学：从“瞬时快照”到“上下文状态流”

在旧方案中，我们最大的痛点是**特征稀疏**——计算步骤（Compute）没有坐标信息，导出步骤（Export）没有计算负载信息，导致特征矩阵里充斥着 `0`。这不仅浪费了 LSTM 的计算能力，也丢失了上下文关联。

**新方案的核心变革在于引入了“状态保持（Stateful）”机制**。我们将特征提取看作一个**状态机**：

- 一旦在日志序列中看到了坐标（BBox），这个坐标就会被“记忆”在上下文寄存器中。
- 后续的计算、归档等步骤，虽然日志本身没有写坐标，但特征提取器会自动将“记忆”中的坐标填入特征向量。
- **意义**：这模拟了真实的计算机内存行为——数据被加载到内存后，CPU 对其进行处理时，数据的空间属性依然存在。

## 2. JSON 结构化日志格式说明 (`simulated_logs.json`)

新版日志为了实现“流量伪装（Traffic Camouflage）”，在结构上抹平了正常与恶意的显式差异。

### 2.1 根对象 (Job Level)

每一个 JSON 对象代表一个完整的计算任务：

- **`id`**: 任务唯一标识符（UUID）。
- **`label`**: **0** (正常业务) 或 **1** (恶意攻击)。这是监督学习的 Ground Truth。
- **`logs`**: 一个包含多条事件（Event）的列表，按时间发生顺序排列。

### 2.2 事件对象 (Event Level)

列表中的每一项代表一行日志，结构如下：

JSON

```
{
  "seq": 3,                          // 序列号，确保时序绝对正确
  "ts": "2025-12-05T10:30:05.123",   // ISO 8601 时间戳
  "act": "DATA_PROCESS",             // 动作类型 (枚举值)
  "meta": {                          // 元数据 (参数池，不同动作包含不同字段)
    "step": 1,
    "load": 0.75,
    "bbox": [37.2, 39.2, 120.7, 122.7], // 仅在 DATA_ACCESS 出现
    "size_kb": 15000                    // 仅在 RESULT_ARCHIVE 出现
  }
}
```

**混淆设计亮点**：

- **动作同质化**：无论黑白，动作序列都是 `JOB_START` -> `DATA_ACCESS` -> `DATA_PROCESS` (多次) -> `RESULT_ARCHIVE`。
- **参数隐蔽化**：恶意任务不再使用特殊的 Action Name，而是隐藏在 `meta` 数据的统计分布（如坐标位置、时间间隔）中。

## 3. 特征提取方案详述 (`lstm_feature_data.json`)

我们将每个任务转化为一个 **`Sequence Length × Feature Dim`** 的矩阵。

- **最大序列长度 (`MAX_SEQ_LEN`)**: 10（不足补 0，超过截断）。
- **特征维度 (`Input Size`)**: 5维。

以下是这 5 个特征维度的具体提取逻辑与物理含义：

### 第 1 维：动作编码 (Action ID)

- **类型**：离散值 (Discrete)。
- **提取逻辑**：直接查表映射。
  - `PAD` (填充) -> 0
  - `JOB_START` -> 1
  - `DATA_ACCESS` -> 2 (获取数据)
  - `DATA_PROCESS` -> 3 (计算/处理)
  - `RESULT_ARCHIVE` -> 4 (导出结果)
- **科研意义**：这是 LSTM 识别“行为阶段”的锚点。比如，LSTM 会学习到“在 Action=3 的阶段，如果时间差很大，且 Action=2 曾指向敏感区，则是攻击”。

### 第 2 维：时间增量 (Time Delta)

- **类型**：连续值 (Continuous)，经过对数平滑。
- **提取逻辑**：
  1. 计算当前日志时间戳与**上一条**日志时间戳的差值（秒）：`delta = current_ts - last_ts`。
  2. **Log1p 变换**：`feature = np.log1p(delta)`。
- **为什么要取对数？**
  - 真实场景中，步骤间隙可能是 0.1秒（初始化），也可能是 1000秒（复杂计算）。直接使用原始秒数会导致数值跨度过大，难以归一化。`log1p` 能将长尾分布拉回正态分布，利于模型收敛。
- **科研意义**：捕捉**“计算复杂度”**。恶意攻击（求解流体方程）通常比简单的统计分析（正常任务）耗时更长、更稳定。

### 第 3 & 4 维：上下文经纬度 (Context Lat/Lon)

- **类型**：连续值 (Continuous)，归一化到 [0, 1]。
- **提取逻辑 (状态保持核心)**：
  - 初始化 `ctx_lat = 0`, `ctx_lon = 0`。
  - **遍历日志序列**：
    - 如果遇到 `DATA_ACCESS` 且包含 `bbox` 参数：
      - 计算中心点：`lat = (min_lat + max_lat) / 2`
      - 更新寄存器：`ctx_lat = lat`, `ctx_lon = lon`
    - 如果遇到其他动作（如 `DATA_PROCESS`）：
      - **直接使用寄存器中的值**。
- **科研意义**：这是解决特征稀疏的关键。它告诉模型：“虽然这一步是单纯的 CPU 计算，但它处理的是**位于 (38.25, 121.75) 的数据**”。这使得模型能将“敏感区域”与“高负载计算”这两个特征关联起来。

### 第 5 维：多态数值度量 (Polymorphic Value Metric)

- **类型**：连续值 (Continuous)，归一化。

- 提取逻辑：

  这是一个“复用”的维度，根据 Action 的不同代表不同含义：

  - 如果是 **处理阶段** (`load` 存在)：提取 `meta['load']` (CPU负载/计算强度)。
  - 如果是 **归档阶段** (`size_kb` 存在)：提取 `meta['size_kb']` (文件大小)。
  - 其他情况：填 0。

- **科研意义**：

  - 对于攻击者，为了掩人耳目，可能会把大计算量拆分成多次小的 `DATA_PROCESS`。
  - 对于输出，物理模拟生成的网格数据（NetCDF）通常比统计报表（CSV）大得多。
  - LSTM 会结合第 1 维 (Action ID) 自动学会：当 Act=3 时关注负载，当 Act=4 时关注文件大小。

## 4. 总结：为什么这套方案更“高级”？

1. **信息密度极高**：通过状态保持，我们消除了 60% 以上的无效 `0` 值，每一行特征向量都携带了完整的上下文信息（做了什么、花了多久、在哪里做、耗费多少资源）。
2. **抗混淆能力强**：即使攻击者将一次长计算拆分成 10 次短计算（试图降低单步 Time Delta），LSTM 依然能通过 Sequence 特征（连续 10 个 Action=3 且 Context Lat 指向敏感区）识别出异常。
3. **真实感**：这套方案生成的特征分布存在重叠（Overlap），例如正常任务偶尔也会查大文件，偶尔也会在敏感区边缘活动。这种“不完美”的数据正是训练高鲁棒性模型所必需的。
