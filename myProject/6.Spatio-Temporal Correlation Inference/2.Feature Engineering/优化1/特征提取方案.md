这非常敏锐。您指出的两个问题（特征稀疏、样本太易区分）确实是初级模拟数据常犯的错误，这会导致模型训练毫无意义（因为简单的 `if action == 'COMPUTE_STEP'` 就能达到 100% 准确率）。

为了让 LSTM 真正发挥作用，我们需要构建一个**“高混淆度”**的数据集，让恶意行为在表面上与正常行为几乎一致，只有通过**时序上下文**和**多维特征的非线性组合**才能区分。

以下是针对您提出的问题的深度分析与解决方案：

### 问题分析与解决策略

**1. 针对“特征稀疏（太多 0）”的解决：**

- **原因**：之前的逻辑是“一步一特征”。例如 `DATA_FETCH` 步有坐标但没有耗时，`COMPUTE` 步有耗时但没有坐标，导致矩阵中出现大量空洞。
- **策略**：引入 **“状态保持（Stateful）”** 特征提取。
  - **坐标延展**：一旦 `DATA_FETCH` 设定了经纬度，后续的计算和导出步骤应该**继承**这个坐标上下文（因为计算是针对该区域数据的）。
  - **填充策略**：不再单纯补 0，而是保留上下文状态，使每一行特征向量都尽可能稠密。

**2. 针对“Normal 与 Malicious 太容易分辨”的解决：**

- **原因**：之前的攻击使用了独有的 Action (`COMPUTE_STEP`) 和任务类型 (`Physics`)，这相当于直接把答案写在了脸上。
- **策略**：**“流量伪装（Traffic Camouflage）”**。
  - **统一词汇表**：攻击任务和正常任务使用**完全相同**的 Action 集合（如都叫 `DATA_PROCESS`）。
  - **行为重叠**：正常任务也加入随机的“多步处理”和“长耗时”，模拟复杂科研计算。
  - **差异微观化**：攻击的唯一破绽在于**“特定的坐标组合”**与**“该坐标下的特定计算耗时/产出比”**。例如，查询了A区域（禁区外围），却进行了极高强度的计算（插值推断），这才是 LSTM 需要捕捉的隐式关联。

#### 时空关联日志特征提取方案与实现细节

#### 1. 核心设计理念：上下文状态流 (Contextual State Flow)

为了解决原始日志中“动作-参数”分离导致的特征矩阵稀疏问题，本方案放弃了“所见即所得”的提取方式，转而采用**状态机（State Machine）**模式。

- **逻辑**：一个计算任务是一个连续的过程。如果在第 2 步定义了“我正在处理经纬度 (38, 121) 的数据”，那么第 3、4、5 步的 `DATA_PROCESS` 动作虽然没有显式参数，但实际上都在处理这个坐标的数据。
- **实现**：特征提取器维护一组 `ctx_` 变量（Lat, Lon, Area）。当遇到 `DATA_ACCESS` 时更新这些变量；当遇到后续步骤时，**复用**这些变量填充特征向量。这消除了 `Lat/Lon` 列在计算步骤中的 0 值，使得 LSTM 在每一步都能看到完整的空间上下文。

#### 2. JSON 日志格式变更说明 (Schema Update)

新的 JSON 旨在最大化混淆度，不再包含 `COMPUTE_STEP` 等显式恶意关键词。

| **字段**       | **说明**                                                     | **正常 vs 恶意差异**                                         |
| -------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| `act` (Action) | 统一使用 `JOB_START`, `DATA_ACCESS`, `DATA_PROCESS`, `RESULT_ARCHIVE`。 | **无差异**。双方共享完全相同的动作序列，迫使模型通过参数分布来判断。 |
| `meta.bbox`    | 数据的经纬度边界框。                                         | **分布差异**。恶意任务的 bbox 中心点会统计性地聚集在 `(38.25, 121.75)` 周围，但会有随机扰动。 |
| `ts` (Time)    | 时间戳。                                                     | **时序差异**。恶意任务的 `DATA_PROCESS` 步骤之间的间隔（Delta Time）通常略长且稳定（模拟物理计算），正常任务波动较大。 |
| `meta.size_kb` | 导出文件大小。                                               | **数值差异**。恶意任务（网格数据）的大小分布通常比正常统计数据（CSV）更集中且偏大。 |

#### 3. 特征向量维度详解 (6维)

每一行日志被转化为一个 6 维稠密向量：

1. **Action ID (离散)**: 动作的类别索引。
2. **Log(Delta Time) (连续)**: 距离上一步的时间差的对数值。使用对数是为了压缩时间尺度的差异（毫秒级到分钟级）。
3. **Context Lat (连续)**: **[关键]** 当前步骤关联的纬度中心。利用“状态流”填充，确保计算步骤也有坐标特征。
4. **Context Lon (连续)**: **[关键]** 当前步骤关联的经度中心。同上。
5. **Context Area (连续)**: **[新增]** 处理区域的面积大小。攻击者为了隐蔽，可能会查询大区域但只计算小部分，或者反之，这个特征有助于模型学习“区域大小-计算时长”的比例关系。
6. **Value Metric (连续)**: 资源量度。在处理步代表 `load_avg`，在归档步代表 `file_size`。这是一种多态特征，LSTM 会根据 Action ID 自动理解该维度的含义。

#### 4. 预期效果

通过这套方案生成的 `lstm_feature_data.json` 将具有以下特性：

1. **稠密性**：除了 Padding 部分，有效序列内部几乎没有 0 值，信息量极大。
2. **隐蔽性**：单看某一行特征（例如 `[3, 0.5, 0.8, 0.8, 0.2, 0.6]`），人类无法判断是好是坏。只有当模型看到整个序列，发现“这个坐标组合”配合“这种计算时长”出现的概率在统计上属于异常模式，才能给出高置信度的判断。这正是 Deep Learning 存在的意义。
