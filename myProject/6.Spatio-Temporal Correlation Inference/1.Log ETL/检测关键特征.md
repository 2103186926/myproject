这是一个极具挑战性但也更符合现实攻防场景的需求。真正的攻击者绝不会在日志里打印“我正在攻击禁区”，他们会伪装成普通的科研计算任务。

为了满足您的要求，我重构了 `log_generator.py`。

**优化核心策略：**

1. **去标签化（Stealth）**：移除了所有“Restricted Area”、“Attack”、“Inferred”等敏感关键词。所有日志看起来都是标准的 `global_ocean_analysis` 数据处理。
2. **行为隐写（Behavioral Steganography）**：
   - **正常任务**：表现为“查询 -> 统计/聚合 -> 结果保存”。计算快，无迭代。
   - **攻击任务**：表现为“查询（范围略大） -> 网格插值 -> 迭代求解（耗时） -> 结果保存”。
   - **差异点**：差异隐藏在**坐标数值（空间特征）**、**日志条数序列（时序长度）**、**时间差（计算耗时）**以及**处理步骤的术语（迭代 vs 聚合）**中。



### 特征工程文档：正常日志 vs 恶意行为

这份文档将指导您后续如何从上述“通用格式”的日志中提取特征，输入给 LSTM 模型。因为日志没有显式标签，我们必须依赖**序列行为**和**数值分布**。

#### 核心差异对照表

| **特征维度**                 | **正常科研行为 (Normal)**                                    | **时空关联推理攻击 (Malicious)**                             | **对应文档依据**                      |
| ---------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ | ------------------------------------- |
| **空间特征 (Spatial)**       | **随机分布** 坐标随机落在太平洋、大西洋等公海区域。 Lat/Lon 变化大。 | **固定/聚集分布** 坐标总是围绕特定的敏感坐标 (38.25, 121.75) 波动。 即使是随机扰动，也呈现聚类特征。 | 模拟域的巧合定位                      |
| **计算模式 (Computational)** | **单次聚合** `Running aggregation` -> `Complete` 中间没有循环步骤。 | **迭代求解** `Iterative solver` -> `step=10` -> `step=50`... 存在明显的循环日志序列。 | 物理模型的工具化使用                  |
| **时间特征 (Temporal)**      | **短时爆发** 数据加载后，计算过程通常在 2-3秒内完成。        | **持续长时** 由于迭代求解，计算阶段持续 10-20秒甚至更长。 日志时间戳跨度显著拉大。 | 计算密集型优化循环 (参考梯度攻击特征) |
| **输入/输出语义 (I/O)**      | **输入输出一致** 查询 BBox 和处理逻辑通常匹配。              | **环带模式** 查询的是 Buffer Zone，但处理逻辑是 Interpolation (插值填补)。 | 边界条件的环带模式                    |

#### 建议提取的 LSTM 动态时序特征

为了让 LSTM 检测到上述差异，建议在第二步（JSON结构化与特征提取）中重点提取以下向量：

1. **关键词嵌入 (Keyword Embedding)**:
   - 将日志中的关键动词映射为 ID：`Fetch`, `Aggregation`, `Solver`, `Interpolation`, `Export`.
   - **预期**: 攻击序列会包含 `Fetch -> Interpolation -> Solver -> Solver -> Solver -> Export` 的特定序列模式。
2. **坐标数值归一化 (Coordinate Normalization)**:
   - 从 `Query BBox` 中提取 `[lat_min, lat_max, lon_min, lon_max]`。
   - **预期**: LSTM 会学习到 `[37.5, 39.0, 121.0, 122.5]` 这一组数值组合具有高风险权重。
3. **日志条目时间差 (Inter-arrival Time)**:
   - 计算当前日志与上一条日志的时间差 `Delta_T`。
   - **预期**: 正常任务的 `Delta_T` 序列方差小；攻击任务在 `Solver` 阶段会有连续的、较大的 `Delta_T`（模拟计算耗时）。
4. **资源/步骤计数 (Step Count)**:
   - 统计一个 `Job-ID` 下日志的总行数。
   - **预期**: 攻击任务由于包含迭代日志，总行数通常多于正常任务。

通过这种方式，即使没有“Attack”关键字，LSTM 也能通过**“特定的坐标输入 + 迭代计算序列 + 长耗时”**这一组合特征，准确识别出恶意行为。
